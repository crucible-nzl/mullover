§ 11 · Notes for the build

How this page will be wired in production.

The admin portal in production lives at admin.counsel.day, served from the same Hetzner box as the API but routed by Caddy to a separate FastAPI router. Operator access is gated by Auth0, configured with a separate "Operators" connection that only whitelisted email addresses can sign in through, plus hardware-key WebAuthn as a mandatory second factor enforced by an Auth0 Multi-Factor rule. End-user authentication flows for the public product also run through the same Auth0 tenant, but the operator role is governed by a separate Auth0 Application and a custom claim in the JWT.

Every figure on this page is computed on read from the live Postgres database using read-only credentials with a row-level security bypass scoped only to aggregate queries (per-decision content is never read). Charts render via Chart.js (industry-standard CMS chart library); the admin portal is deliberately decoupled from the customer-facing editorial palette, since operator ergonomics and pattern recognition matter more than visual continuity for an internal tool. Secrets (Anthropic API key, Stripe, Auth0, Brevo, Postgres) live in Infisical (self-hosted on the same Hetzner box); the Verdict AI section above is the operator surface for binding, rotation, and audit.

The page auto-refreshes every five minutes; a manual refresh button sits in the period bar. Activity-feed events stream in via Server-Sent Events when the page is open. Nothing on this page leaves the operator's session: no telemetry to third-party analytics, no screenshots cached, no public URL.

What is intentionally not on this page (for v0.1): per-decision content inspection (privacy by design), prompt-template editor, customer-support ticket queue with SLA timers, A/B test reporting, cohort retention curves. The user, subscription, and SKU management tools added in this iteration cover the standard SaaS-admin top level; the listed deferrals are their own future tabs.
